### 프로젝트 제출

- 프로젝트 완성하고 코드 제출하기 위해 전날 모여서 밤을 샜다.  
    - 역할 분담을 크게 **데이터 파이프라인 제작 / 웹 서비스 제작 / AI 모듈** 이렇게 나누었었다.  
    - 그래서  
        - 데이터 ↔ 웹서비스 (DB에 잘 적재되는지)  
        - 웹서비스 ↔ RAG (항목 잘 찾아주는지)  
      이런 식으로 계속 확인했다.
    - README 작성했다.  
        - 나는 README 작성도 중요하다고 생각해서 팀원들에게 각자 하면서 고민한 지점들을 적어달라 했다.  
        - 다들 잘 작성해준 것 같다.  
        - **문제 정의를 어떻게 잘 표현할 것인가**를 고민했다.  
            - 우리는 문제를 잘근잘근 쪼개고 다시 합쳐서 추상화하는 게 좋다고 알고 있다.  
            - 하지만 이걸 처음 보는 사람에게 보여줄 때는 그 선을 적당히 맞춰야 한다고 생각했다.  
            - 적절한 사례와 함께 설명하면, 잘게 나눈 문제에 대해 더 공감하게 설명할 수 있지 않을까 생각했다.

### 실제 사례 기반 QA

- 인터넷에 있는 실제 사례를 가지고 서비스 QA를 진행했다.
    - 실제 견적서를 가져와 테스트해보니, 없는 정보도 있고 글자를 가끔 제대로 인지하지 못하는 경우도 있었다.
- 하지만 이건 우리가 정한 문제 정의에 해당하지 않았다.  
    → 그래서 견적서를 하나하나 만들면서 테스트했다. (굉장히 노가다)

- 몇몇 사례에 대해서는 잘 작동하는 것을 확인했다.  
  나름 만족스러웠다.

### 가장 아쉬웠던 점 – Transform 단계

- 프로젝트 하면서 가장 아쉬웠던 것은 **transform 단계**였다.
- 가장 큰 문제는:
    - DB에 저장된 부품명과 실제 견적서 부품명이 **같은 것이지만 다른 이름을 가지면 인식하지 못한다는 점**이었다.

- 이것도 누군가 잡고 깊게 팠으면, transformation 단계에서 훨씬 풍부하게 말할 수 있었을 것 같다. 나 포함해서 왜 아무도 안 했을까?
    - 딱 봤을 때는 해답이 잘 안 보이는 문제라서 그래서 아무도 시도하지 않았던 걸까… 내일 한번 얘기해봐야겠다.

### 발표 내용에 대한 고민

- 옆 팀이랑 얘기하면서 “발표에 데이터 파이프라인 오류가 났을 때 대처법을 넣는 게 맞을까?”라는 얘기를 했다.
- 데이터 엔지니어링에서 중요한 부분이니까 당연히 넣어야 한다는 생각했다. 하지만 이건 너무 당연한 거라 굳이 얘기해야 하냐는 논리도 있었다.
- 맞는 말 같았다. 프로젝트를 항상 끝내고 나면 내가 한 게 별게 아니라는 기분이 드는 것 같다. (실제로 맞기도 하지만)
- 하지만 실제로 데이터 파이프라인을 매일 돌려보며 오류를 마주하고, 그걸 하나씩 풀어내는 과정은 별거 않게 보여도 큰 경험이고 생각했다.
