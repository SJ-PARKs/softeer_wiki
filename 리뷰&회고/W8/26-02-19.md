# 오늘 리뷰

### 연휴동안 작업물 공유
- 연휴 동안 진행한 작업들을 공유하는 시간을 오전에 가졌다.
- 제일 우려스러웠던 AI 관련해서는 정비 지침서와 크롤링한 정비 관련 글들을 기반으로 RAG 구조를 구성했다.
- 담당하신 분이 LLM을 만들면서, 답변 퀄리티 문제로 인해 LLM으로 과잉정비를 진단하는 것에 대해 자신이 없다고 말씀하셨다.
→ 그 마음은 충분히 이해가 갔다. 다만 단순 부품비만으로 판단하는 것은 약하다는 의견이 있었고, 그래서 나머지 팀원들이 계속 시도해보자고 설득했다.
- 하지만 해당 AI가 해결하려는 문제가 명확하지 않다고 느꼈다. 다소 추상적이라는 생각이 들어, 다시 문제 정의 단계로 돌아갔다.
- 그 결과, “부품 수리 대신 통교체”하는 사례들을 모아서 해결할 수 있는지 검토해보았다. 다행히 어느 정도 성능이 나왔던 것 같다.
- 또한 UI 상에서 과잉정비에 대해 하나의 ‘총평’을 내려줄 것인지, 아니면 과잉정비를 쪼갠 여러 개의 세부 문제들에 대해 각각 판단하고 보여줄 것인지 팀 내에서 고민했다.
→ 이와 관련해 부품비와 공임비가 서로 연관이 있는지부터 논의했지만, 결국 둘을 합쳐서 판단해주는 명확한 기준이 애매해 개별적으로만 보여주기로 결정했다.
- 비대면으로 작업하면 개인 시간이 많아져 작업을 치고 나갈 수 있는 장점이 있는 것 같다. 반면 대면의 경우에는 서로의 진행 상황을 실시간으로 공유할 수 있어 더 안정적인 느낌이 드는 것 같다.

### 데이터 파이프라인 점검
데이터 파이프라인을 점검하면서 손봐야 할 부분이 생각보다 너무 많다는 느낌을 받았다. 따지고 보면 한도 끝도 없을 것 같았다.   
그래서 우선 아래 사항부터 정리해서 논의했다.
- 만약 extract한 데이터가 계속 실패한다면, 전날 데이터를 그대로 copy할 것인지?
- 전날 데이터를 copy했을 때, 일정 비율(예: n% 이상) 실패하면 코드를 다시 점검하도록 자동화 로직을 짜기로 했다.

### 데이터 파이프라인 구축
- 코드 스쿼드 계정으로 데이터 파이프라인을 migration하는 과정을 진행했다.
- 이 과정에서 IAM 관련 권한이 중간중간 막혀 꽤 애를 먹었다. 하지만 덕분에 IAM 체계에 대해서는 오히려 많이 공부하게 된 것 같다.
