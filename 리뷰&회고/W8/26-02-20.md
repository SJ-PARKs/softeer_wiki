# 오늘 리뷰

### 피드백

멘토님 피드백을 받으면서 부족한점과 발표 전략 같은것을 구상했다.

## **웹 UI**

- 앞페이지에 딱보고 뭘하려는 건지 알아야한다. 설명 같은 밑에 있는 글씨 너무 작고 이해가 안된다.  
  말이 많다고 좋은게 아니다.  

- 이 서비스를 사용하는 사람들은 어떻게 해결 하는지는 관심없다. 가장 중요하게 생각하는 것은 **이게 과잉정비인지 아닌지**이다.  

- 결과 페이지도 너무 짬뽕. 사람들이 제일 궁금해하는 결론이 맨위에 나와야한다. 그 밑에 상세정보들.

- 또 이거의 경우에도 문제가 있는경우 / 없는경우 쪼갤 수 있다는 것도 깨달았다.  
  문제가 있으면 당연히 문제의 상세가 궁금해할 것이고 문제가 없는 경우에도 의심이 아직 남아있고 찜찜할것이다.  
  그래서 뭐뭐가 구체적으로 문제가 없는지를 각각에 대해 설명을 듣고 싶어한다.  

    -> 이 부분은 놓쳤던거 같다. 결국 사람에 대한 이해도가 중요하다는 것을 꺠달았다. 

    -> 결국에는 문제정의랑 또 직결된다. 우선 문제정의를 한 각각에 대해서 UI 두개를 따로 그려보고 합쳐보는게 그냥 대놓고 합치는것보다 훨씬 좋다.    
-> 피드백 직후 회의하면서 발표할 때는 페르소나를 정하고 실제 그 사례가지고 작동하는거를 보여주는 방향으로 가기로 했다.   
-> 모든 문제를 다 푼다는 말이 안된다. 각 문제에 대해서 60%씩만 푸는것도 되게 멍청하다. 하나의 시나리오를 짜고 그 모듈의 성능을 향상하는데 집중을 했다고 풀어야한다.  
그리고 당연히 문제가 있을 때를 가정하고 얘기하고, 없을때도 보여주면 더 풍부해질것이라는 생각됐다.
- 지금 우리가 각각의 문제를 풀고 있는데 이 각각의 문제를 합쳤을 때 별로 value가 없을 수 있다. 엣지 케이스에 맞춰서 생각해보자.

## **데이터 파이프라인**

- 부품비 하나에 대해서도 고려할게 많다. 얼마나 효율적으로 잘 긁어올것인가.  
  정확도부터 시작해서 이름 안맞는 정규화를 어떻게 할것인가. 어떤 주기로 update 할것인가  

    -> 뭔가 우리 시스템이 크게 복잡하지 않아 발표때 어떻게 잘 말할지 고민..  
   최선의 방법이 아닌 차선을 선택한 느낌이 있지만 잘 정리해서 발표때 녹여야겠다고 생각한다.  

    -> 지금 각각에 대해서 따로 정리해보자. 현재 부품비, 공임비, 정비 사례가 있고  
   각각에 대해서 어떤 툴을 사용했는지 말고 큰 추상화 관점에서 정리해보자. 
   정비사례에 대해서도 팀원에게 잘 말해서 정리해달라고 해야겠다. 언제, 어떻게 할건데 어디서 수집을 해서 어떻게 정규화 할건지, 자동화는 할건지  

- 만약 계속 가져온다면 캐시 시스템으로, 일주일에 한번 적재한다면 그냥 일반 rdb로.  
  use case에 따라 완전히 다른 시스템이 될 수 있다.  

- 데이터 api (value를 만드는것) vs transformation (단순히 정제, 파일 -> 파일)  

- 발표할 때 시스템중 제일 고장 나기 쉬운 곳을 찝고 거기가 에러가 난다면 어떻게 retry를   어떻게 대응할것인가까지 말해봐야겠다.
